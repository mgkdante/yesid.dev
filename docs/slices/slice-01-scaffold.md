## Objective

Initialize a SvelteKit 2 project with Bun, TypeScript, Tailwind CSS (using brand tokens), Vitest, Playwright, and a GitHub Actions CI pipeline. The project builds, renders a blank branded page, passes one smoke test, and CI is green.

## Technical Spec

### Step 1: Initialize SvelteKit with Bun

The repo is not empty. Existing files (brand/, docs/, CLAUDE.md, README.md, etc.) must be preserved.

Create the SvelteKit project in a temporary directory, then merge:

```bash
cd /tmp
bunx sv create yesid-temp --template minimal --types ts
```

Copy the generated SvelteKit files into the project root:
- `src/` contents (app.html, app.d.ts, routes/+page.svelte, etc.)
- `static/` directory
- `svelte.config.js`
- `vite.config.ts`
- `tsconfig.json`
- `package.json` (merge carefully: keep name as "yesid-dev", add generated dependencies)

Remove the temp directory when done.

Remove `src/.gitkeep` (src/ now has real files).

### Step 2: Install dependencies with Bun

```bash
bun install
```

Then add the additional packages:

```bash
bun add -d tailwindcss @tailwindcss/vite
bun add -d @sveltejs/adapter-vercel
bun add -d vitest @testing-library/svelte @testing-library/jest-dom jsdom
bun add -d @playwright/test
```

Note: check if SvelteKit's `sv create` already installs some of these. Don't duplicate.

### Step 3: Configure Tailwind CSS

Create `src/app.css`:
```css
@import 'tailwindcss';
```

Import it in `src/routes/+layout.svelte`:
```svelte
<script>
  import '../app.css';
</script>

<slot />
```

Update `vite.config.ts` to include the Tailwind plugin:
```ts
import { sveltekit } from '@sveltejs/kit/vite';
import tailwindcss from '@tailwindcss/vite';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [
    tailwindcss(),
    sveltekit()
  ]
});
```

Extend the brand config. The Tailwind v4 approach uses CSS-based config, but if v3 is generated by sv create, use `tailwind.config.js` at project root:

```js
import brandConfig from './brand/tailwind.brand.js';

/** @type {import('tailwindcss').Config} */
export default {
  ...brandConfig,
  content: ['./src/**/*.{html,js,svelte,ts}'],
};
```

**Important:** Check which Tailwind version SvelteKit installs. If Tailwind v4 (CSS-first config), adapt accordingly. Document the decision in the dev log.

### Step 4: Resolve duplicate Tailwind config

Delete `brand/tailwind_brand.js`. Keep `brand/tailwind.brand.js` as the canonical brand config.

Why: `tailwind.brand.js` uses standard npm naming (dots as separators). The underscore version is a duplicate.

### Step 5: Brand tokens integration

Copy `brand/tokens.css` to `src/lib/styles/tokens.css`.

Import it in `src/app.css` (before Tailwind):
```css
@import '$lib/styles/tokens.css';
@import 'tailwindcss';
```

This makes all CSS custom properties (--brand-primary, --bg-primary, --text-primary, etc.) available globally.

Keep `brand/favicon.svg` as the source asset (do not delete it).

### Step 8: Vercel adapter

Install if not already:
```bash
bun add -d @sveltejs/adapter-vercel
```

Update `svelte.config.js`:
```js
import adapter from '@sveltejs/adapter-vercel';
import { vitePreprocess } from '@sveltejs/vite-plugin-svelte';

/** @type {import('@sveltejs/kit').Config} */
const config = {
  preprocess: vitePreprocess(),
  kit: {
    adapter: adapter()
  }
};

export default config;
```

### Step 9: Configure Vitest

Update `vite.config.ts` to include Vitest config:
```ts
import { sveltekit } from '@sveltejs/kit/vite';
import tailwindcss from '@tailwindcss/vite';
import { defineConfig } from 'vitest/config';

export default defineConfig({
  plugins: [
    tailwindcss(),
    sveltekit()
  ],
  test: {
    include: ['src/**/*.{test,spec}.{js,ts}'],
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/tests/setup.ts']
  }
});
```

Create `src/tests/setup.ts`:
```ts
import '@testing-library/jest-dom/vitest';
```

Add to `package.json` scripts:
```json
{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest",
    "test:e2e": "playwright test"
  }
}
```

### Step 10: Configure Playwright

Create `playwright.config.ts` at project root:
```ts
import type { PlaywrightTestConfig } from '@playwright/test';

const config: PlaywrightTestConfig = {
  webServer: {
    command: 'bun run build && bun run preview',
    port: 4173,
    reuseExistingServer: !process.env.CI
  },
  testDir: 'tests',
  testMatch: '**/*.spec.ts'
};

export default config;
```

Install Playwright browsers:
```bash
bunx playwright install --with-deps chromium
```

Remove `tests/.gitkeep` (tests/ will have real files in slice 08).

### Step 11: Write smoke test

Create `src/routes/+page.svelte` (replace whatever sv create generated):
```svelte
<div data-testid="app-root" class="min-h-screen bg-[var(--bg-primary)] text-[var(--text-primary)] font-body">
  <main class="max-w-5xl mx-auto px-6 py-20">
    <h1 class="text-4xl font-bold font-heading">
      yesid<span class="text-brand-primary">.</span>
    </h1>
    <p class="mt-4 text-[var(--text-secondary)]">
      Data infrastructure that moves.
    </p>
  </main>
</div>
```

Create `src/routes/+page.test.ts`:
```ts
import { describe, it, expect } from 'vitest';
import { render, screen } from '@testing-library/svelte';
import Page from './+page.svelte';

describe('Home page smoke test', () => {
  it('renders the app root', () => {
    render(Page);
    expect(screen.getByTestId('app-root')).toBeInTheDocument();
  });

  it('renders the wordmark', () => {
    render(Page);
    expect(screen.getByText('yesid')).toBeInTheDocument();
  });

  it('renders the tagline', () => {
    render(Page);
    expect(screen.getByText('Data infrastructure that moves.')).toBeInTheDocument();
  });
});
```

In `src/app.html`, add `class="dark"` and `data-theme="dark"` to the `<html>` tag:

### Step 13: GitHub Actions CI

Create `.github/workflows/ci.yml`:
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Bun
        uses: oven-sh/setup-bun@v2
        with:
          bun-version: latest

      - name: Install dependencies
        run: bun install

      - name: Sync SvelteKit
        run: bunx svelte-kit sync

      - name: Type check
        run: bunx svelte-check --tsconfig ./tsconfig.json

      - name: Unit tests
        run: bun test

      - name: Build
        run: bun run build
```

Note: E2E tests (Playwright) are added to CI in slice 08 after all pages exist. CI starts with unit tests and type checking only.

### Step 14: Cleanup

Remove `scripts/.gitkeep` (keep the directory if empty, or remove if nothing will go there in this slice).

```bash
tree -I "node_modules|.git|.remember|bun.lockb|.svelte-kit|.vercel|dist|build|*.log|.DS_Store" --charset ascii > tree.txt
```

On Windows if tree doesn't support -I, generate manually.

## Learn

**SvelteKit project structure:** The `src/routes/` directory maps to URL paths. `+page.svelte` is a page, `+layout.svelte` wraps pages. `src/lib/` is for shared code. Think of it like SQL: routes are "views" that present data, lib/ is the "schema" where the logic lives.
- Read: https://svelte.dev/docs/kit/project-structure

**Tailwind config extension:** `tailwind.brand.js` defines your brand's design tokens as Tailwind utilities. Extending it means every component can use `text-brand-primary` or `bg-dark-surface` without remembering hex codes. Like creating a data dictionary for your UI.
- Read: https://tailwindcss.com/docs/configuration

**Vitest:** Fast test runner built for Vite (which SvelteKit uses). `@testing-library/svelte` renders components in a simulated DOM so you can assert "does this render what I expect?" Think of tests as CHECK constraints on your UI.
- Read: https://vitest.dev/guide/

**GitHub Actions:** A YAML file that tells GitHub "on every push, run these commands." If any step fails, the push is flagged. Like a database trigger that validates data before a commit is accepted.
- Read: https://docs.github.com/en/actions/quickstart

**Bun:** A JavaScript runtime like Node.js but faster. Same package.json, same code. Main differences: `bun install` is faster, `bun.lockb` is binary, and some APIs are Bun-native.
- Read: https://bun.sh/docs
