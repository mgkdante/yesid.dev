## How tokens flow

```
packages/tokens/tokens.json
       │
       ├─→ apps/web/src/lib/styles/tokens.css           (generated; runtime CSS)
       ├─→ apps/web/src/app.css  @theme block           (generated; Tailwind utilities)
       ├─→ apps/web/src/lib/motion/tokens.ts            (generated; TS for GSAP at compute time)
       ├─→ /DESIGN.md                                   (generated; agent-readable)
       │
       └─→ Figma Variables (this file)                  (manual push — see "Sync flow" below)
```

The first 4 outputs are auto-generated by `bun run --cwd packages/tokens build` and parity-tested against `tokens.json`. The 5th (Figma) is agent-driven via the `figma-remote` MCP — not a CI step.

## Sync flow

To re-sync after editing `tokens.json`:

```bash
bun run --cwd packages/tokens build              # regenerate code outputs (CSS / TS / Markdown)
bun run packages/tokens/scripts/push-to-figma.ts # prepare Variables payload (stdout JSON)
```

Then in a Claude Code session with figma-remote MCP available:

> "Read the prepared JSON and call mcp__figma-remote__use_figma to create/update each Variable in Figma file Zn8w5sVMwCXhziyv03dpft. Group color/dark/* and color/light/* under dark/light modes; everything else under a single 'value' mode."

The push script is idempotent — re-running updates existing Variables instead of duplicating. To confirm after push:

```bash
# Have Claude write current Figma state to .tmp.figma-export.json (use_figma reads variables)
bun run packages/tokens/scripts/verify-roundtrip.ts
# expect: "OK: <N> variables in parity"
```

## MCP servers

| Server | URL | Auth | Capabilities |
|---|---|---|---|
| **Remote (used)** | `https://mcp.figma.com/mcp` | Figma OAuth | Full read + write to canvas |
| **Desktop (fallback)** | `http://127.0.0.1:3845/mcp` | None (local) | Read-only context |

Connect remote via:

```bash
claude mcp add --scope user --transport http figma-remote https://mcp.figma.com/mcp
# Then run any figma-remote tool to trigger OAuth
```

Both can coexist. Desktop is useful for offline read-only verification; remote is required for any write.

The `Color` collection's two modes (`dark`, `light`) drive theme-switching. Brand colors don't theme-switch by design; they receive the same value in both modes (so every variable in the collection has a value for every mode, as Figma requires).

**Variable name collisions to watch:** `card` exists in both Color (the surface color) and Shadow (`shadow/card`) collections after my prefix-stripping. Lookups must scope to a collection. The push-to-figma logic doesn't trigger this because each collection is created in isolation; the followup component-build code learned this the hard way.

## Round-trip notes

End-to-end round-trip verified on 2026-04-27 with `verify-roundtrip.ts`: **69 expected = 69 actual, exit 0**.

Acceptable serialization deltas (documented per spec § 3.5):

- **`description` HTML-encoded.** Figma stores `'yesid.'` as `&#39;yesid.&#39;` (apostrophes encoded). Verify-roundtrip ignores descriptions.
- **`FLOAT` precision drift.** `0.3` round-trips as `0.30000001192092896`. Float32 storage in Figma. Verify-roundtrip compares mode keys only, not values.
- **Color `STRING` values pass through unchanged.** `color/primary-rgb` (`"224 120 0"`) is preserved as a string.

If a future round-trip surfaces a NEW kind of delta, document it here.
