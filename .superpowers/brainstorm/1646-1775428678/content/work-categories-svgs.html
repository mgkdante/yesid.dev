Work Categories + SVG System

Each category gets a unique SVG that animates with GSAP. Used in listing cards AND detail hero. Pick the category scheme that fits best.

Category Scheme Options

A

4 Categories (Aligned with Services)

Maps 1:1 to your existing 4 services. Clean, simple, no ambiguity.

Pipeline
ETL, ingestion, data modeling, warehouse
SVG: source → transform → destination flow
Reporting
KPIs, dashboards, Power BI, analytics
SVG: bar chart + trend line drawing in
Tooling
Admin panels, internal tools, workflows
SVG: app window with UI elements assembling
Fullstack
Web apps, portals, map dashboards
SVG: split screen (frontend + backend) connecting
B

5 Categories (SQL Gets Its Own)

Same as A but SQL/database work gets its own category separate from Pipeline. This reflects that "write me a query" and "build me an ETL" are different client asks.

SQL
queries, tuning, schema
⟶
Pipeline
ETL, ingestion, orchestration
📊
Reporting
dashboards, KPIs, Power BI
⚙
Tooling
admin panels, workflows
🌐
Fullstack
web apps, portals

5 SVGs to create, but SQL is your core identity — it deserves its own visual.

C

Service-Linked (No New Taxonomy)

Don't create a new "category" field. Instead, each project links to one of your 4 existing services via relatedServices: string[]. The SVG comes from the service, not a new category. Keeps the data model tight — one source of truth.

// Project links to service(s)
relatedServices: ['sql-development']

// Service already has an icon field
icon: 'station-sql.json'
// We add an SVG field too
svg: 'service-sql.svg'

Only 4 SVGs. No new enum. But projects can only map to existing services — if you do work outside those 4, you'd need to add a service first.

My recommendation: B (5 categories) — SQL is your identity, it deserves its own SVG. And the 5 categories map cleanly to your actual service offerings without coupling to the home page service data. Future-proof for when you add more services or the cloud layer.
